# jspsych-ado: Adaptive Design Optimization for jsPsych

> A walkthrough of what changes — and what stays the same — when you run an ADO experiment instead of a standard fixed-design one.

---

## The Standard jsPsych Pattern

In a standard jsPsych experiment, you define a trial once and repeat it. Every participant sees the same questions in the same order.

```js
// 1. Import the plugin you need
import jsPsychHtmlButtonResponse from "@jspsych/plugin-html-button-response";

// 2. Define a trial (fixed stimulus, fixed choices)
const trial = {
  type: jsPsychHtmlButtonResponse,
  stimulus: "<p>Which line is longest?</p>",
  choices: ["A", "B", "C"],
};

// 3. Run: every participant gets the same 3 trials
jsPsych.run([instructions, trial, trial, trial, end_screen]);
```

The key property: the stimulus is **fixed at the time it's written/created**. Every participant sees identical questions.

---

## The jspsych-ado Pattern

jspsych-ado mirrors this structure exactly — you still call `jsPsych.run()` with an array — but the trials in that array are generated by `createAdoTimeline()`, which inserts an adaptive loop. Instead of repeating a fixed trial, the experiment picks the next question based on everything the participant has answered so far.

The one new concept is the **controller**: an object that knows how to select the next question (the "design") and update its beliefs after each response. Everything else — stimulus presentation, button layout, data logging — is standard jsPsych.

```
Standard:     instructions → trial → trial → trial → end_screen
                                ↑         ↑         ↑
                            (identical) (identical) (identical)

jspsych-ado:  instructions → [ADO timeline fragment] → end_screen
                                ↑         ↑         ↑
                             design₁   design₂   design₃
                            (optimal) (updated) (updated)
```

## Drop-In Usage: Built-In Delay Discounting Model

For one of the most common cases — delay discounting with the hyperbolic model — jspsych-ado ships a ready-to-run controller. The change from a standard experiment is minimal:

```js
import { initJsPsych } from "jspsych";
import { createAdoTimeline } from "./jspsych-ado/ado/ado_timeline.js";
import { createStanAdoController } from "./jspsych-ado/controllers/stan_ado_controller.js";
import hyperbolicModel from "./jspsych-ado/models/hyperbolic/model.js";
import delayDiscountingTask from "./jspsych-ado/tasks/delay_discounting/task.js";

const jsPsych = initJsPsych();

// Step 1 — Create the adaptive controller (runs in a Web Worker, no server needed)
const controller = await createStanAdoController({
  model: hyperbolicModel,
  task: delayDiscountingTask,
  n_trials: 20,
});

// Step 2 — Build the adaptive timeline fragment
const adoTrials = createAdoTimeline(jsPsych, controller, {
  n_trials: 20,
  response_labels: delayDiscountingTask.response_labels,   // { 0: "SS", 1: "LL" }
  presentation: delayDiscountingTask.presentation,         // stimulus rendering
  choices: delayDiscountingTask.choices,                   // ["SS", "LL"]
  task: delayDiscountingTask.id,
});

// Step 3 — Run: same shape as a standard experiment
jsPsych.run([instructions, ...adoTrials, end_screen]);
//                          ^^^^^^^^^^
//                   replaces [trial, trial, trial, ...]
```

**What changed:** two setup lines (controller + timeline) replaced the repeated `trial` objects.  
**What stayed the same:** `jsPsych.run()`, instructions, end screen, data logging.

## Custom Model Usage

If you have your own cognitive model, you supply a Stan model string and a link function instead of the built-in hyperbolic model. The rest of the call is identical.

```js
import { initJsPsych } from "jspsych";
import { createAdoTimeline } from "./jspsych-ado/ado/ado_timeline.js";
import { createStanAdoController } from "./jspsych-ado/controllers/stan_ado_controller.js";

const jsPsych = initJsPsych();

// --- Define your own model ---

// 1a. Stan source: priors + likelihood for your model parameters
const myStanModel = `
  parameters { real<lower=0> k; real<lower=0> tau; }
  model {
    k   ~ lognormal(-4.0, 1.5);
    tau ~ lognormal(-1.0, 1.0);
    for (n in 1:N) {
      real v_ll  = r_ll[n] / (1.0 + k * t_ll[n]);
      real v_ss  = r_ss[n] / (1.0 + k * t_ss[n]);
      y[n] ~ bernoulli_logit(tau * (v_ll - v_ss));
    }
  }
`;

// 1b. Design grid: the space of possible questions ADO searches over
const myGridDesign = {
  t_ss: [0],
  t_ll: [1, 2, 4, 8, 16, 26, 52],   // delays in weeks
  r_ss: [100, 200, 400],             // smaller-sooner amounts in $
  r_ll: [800],                       // larger-later amount in $
};

// 1c. Link function: P(choose LL | theta, design)
//     Called once per grid point; must be fast.
function linkProb(theta, design) {
  const { k, tau } = theta;
  const v_ll = design.r_ll / (1 + k * design.t_ll);
  const v_ss = design.r_ss / (1 + k * design.t_ss);
  return 1 / (1 + Math.exp(-tau * (v_ll - v_ss)));
}

// --- Wire up (same three steps as before) ---

const controller = await createStanAdoController({
  stanModel: myStanModel,
  grid_design: myGridDesign,
  linkProb,
  n_trials: 30,
});

// Task presentation: how to turn a design into a stimulus
const myPresentation = {
  makeStimulus: (design) =>
    `<p>Would you prefer <strong>$${design.r_ss} now</strong>
       or <strong>$${design.r_ll} in ${design.t_ll} weeks</strong>?</p>`,
};

const adoTrials = createAdoTimeline(jsPsych, controller, {
  n_trials: 30,
  response_labels: { 0: "Now", 1: "Later" },
  presentation: myPresentation,
  choices: ["Now", "Later"],
  task: "my_delay_discounting",
});

jsPsych.run([instructions, ...adoTrials, end_screen]);
```

## What `createAdoTimeline()` Returns

`createAdoTimeline()` returns a plain array of jsPsych trial objects — the same kind of thing you would write by hand. You can spread it into `jsPsych.run()` alongside any other trials.

```js
// The returned array looks like this (schematically):
[
  { type: jsPsychCallFunction, /* initializes the controller */ },
  { type: jsPsychHtmlButtonResponse, /* trial 1 — design chosen by ADO */ },
  { type: jsPsychCallFunction, /* updates posterior, picks next design */ },
  { type: jsPsychHtmlButtonResponse, /* trial 2 */ },
  { type: jsPsychCallFunction, /* ... */ },
  // ... n_trials choice + update pairs
]
```

Each `jsPsychCallFunction` node is invisible to participants; it is the ADO engine updating its beliefs and selecting the next optimal design. The `jsPsychHtmlButtonResponse` nodes are ordinary button-response trials — the same plugin a standard experiment would use.

## The Controller Interface

A controller is a simple object with two async methods. This is for the timeline and the adaptive inference backend:

```js
const controller = {
  // Called once before the first trial.
  // Returns the first design(s) to show.
  start: async (run_context) => ({
    session_id: "abc123",
    trial_index: 0,
    next_design: { t_ss: 0, t_ll: 4, r_ss: 200, r_ll: 800 },
    post_mean: null,
    post_sd:   null,
  }),

  // Called after each trial (or testlet).
  // Receives the participant's response; returns the updated posterior
  // and the next design.
  update: async (trial_data) => ({
    session_id: "abc123",
    trial_index: 1,
    next_design: { t_ss: 0, t_ll: 26, r_ss: 400, r_ll: 800 },
    post_mean: { k: 0.021, tau: 2.3 },
    post_sd:   { k: 0.008, tau: 0.6 },
  }),
};
```

The timeline calls `controller.start()` once at the beginning and `controller.update(trial_data)` after each response. It does not care whether the controller uses Python, Stan-in-WASM, or a lookup table — any object satisfying this interface works.

## Side-by-Side Summary

| | Standard jsPsych | jspsych-ado |
|---|---|---|
| **Trial definition** | Fixed stimulus object | `createAdoTimeline()` generates trials |
| **Question selection** | Authored in advance | Chosen per-trial to maximize information |
| **Inference** | None | Bayesian posterior updated after each response |
| **Entry point** | `jsPsych.run([..., trial, trial, ...])` | `jsPsych.run([..., ...adoTrials, ...])` |
| **Extra setup** | None | Create a controller (2 lines) |
| **Data output** | Standard jsPsych JSON | Standard jsPsych JSON + `post_mean`, `post_sd`, `ado_design` per trial |
| **Backend required** | No | No — Stan runs in a browser Web Worker |
| **Custom models** | N/A | Supply Stan source + `linkProb()` |

## Key Data Fields Added to Each Trial Row

jspsych-ado appends the following fields to the standard jsPsych data row for every choice trial. No post-processing script is needed — they appear in the downloaded JSON alongside `rt`, `response`, and any other standard fields.

| Field | Description |
|---|---|
| `ado_design` | The design object shown on this trial (e.g. `{ t_ss, t_ll, r_ss, r_ll }`) |
| `post_mean_<param>` | Posterior mean of each model parameter after this trial |
| `post_sd_<param>` | Posterior standard deviation of each model parameter after this trial |
| `ado_mutual_info` | Expected information gain for the design shown (design selection diagnostic) |
| `trial_number` | 1-indexed position within the ADO block |
| `choice_label` | Human-readable response label (e.g. `"SS"` or `"LL"`) |
